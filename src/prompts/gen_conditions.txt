You are a test designer. Generate test conditions for ONE API endpoint using Equivalence Partitioning (EP) and Boundary Value Analysis (BVA).

Endpoint specification (JSON):
ENDPOINT_PLACEHOLDER

RULES
1. Use ONLY parameters that appear in the specification: path / query / header parameters and request body property names (top-level keys under requestBody.properties).
2. Use EXACT parameter names. For path template variables use the name without braces (e.g. id, not {id}).
3. Prefer focused conditions: aim for roughly 3–8 conditions per parameter unless the spec is very rich (then at most 12 per parameter).
4. For nested objects or arrays in the body: either (a) add conditions on primitive leaf fields that appear under properties, or (b) one condition on the root field with a minimal valid JSON-compatible object as a single string value ONLY if the prompt format below cannot represent it—otherwise skip exotic shapes.
5. Stay domain-neutral: do not invent business-specific nouns, enums, or semantics not present in ENDPOINT_PLACEHOLDER.

OUTPUT FORMAT
Return a single JSON object with one key "conditions" whose value is an array of objects:
{
  "conditions": [
    {
      "id": "unique_snake_case_id",
      "parameter": "exact_name_from_spec",
      "partition_type": "valid|invalid|boundary",
      "description": "what this condition exercises",
      "values": [ ... ]
    }
  ]
}

VALUES ARRAY
- Each element must be a simple scalar: string, number, boolean, or null where null means “explicit null JSON value” for optional nullable fields ONLY if the spec marks the field as nullable.
- Do NOT put arrays or objects as elements inside "values". If you need several scalars, list them as separate array elements.
- Do NOT use empty string "" as the only value for a condition.
- For “missing optional parameter” or “missing optional body field”, do NOT encode as null in values; the scenario stage will handle omission. Here only model explicit values.

PARTITIONS
- valid: typical valid class, in-range values, valid enums.
- invalid: wrong type, out of range, illegal enum, malformed semantics where obvious from spec.
- boundary: min, max, min-1, max+1 for bounded numerics; minLength/maxLength±1 for strings when given.

ID NAMING
- valid:   {param}_valid_{n}
- invalid: {param}_invalid_{n} or {param}_invalid_type_{hint}
- boundary:{param}_boundary_{n}

Output ONLY one JSON object (no markdown fences, no commentary).
