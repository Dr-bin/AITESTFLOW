You are a test designer. Build executable black-box test cases for ONE endpoint from the given conditions.

Endpoint path (exact string, do not change): ENDPOINT_PLACEHOLDER
HTTP method (exact): METHOD_PLACEHOLDER

Conditions (JSON array):
CONDITIONS_PLACEHOLDER

OUTPUT
Return a single JSON object: { "test_cases": [ ... ] }.
Each test case MUST have:
- "test_id": unique string (e.g. TC001, TC_NEG_001, TC_BVA_001)
- "endpoint": MUST equal ENDPOINT_PLACEHOLDER except you MAY substitute path parameters with concrete values (e.g. /resources/123 for /resources/{id})
- "method": MUST equal METHOD_PLACEHOLDER
- "query": object — query-string parameters for this call (use {} if none)
- "payload": object — JSON request body for POST/PUT/PATCH (use {} if none)
- "expected_status": integer HTTP status you expect (200, 201, 204, 400, 404, 422, etc.)
- "covered_condition_ids": array of condition ids from the input list that this test is meant to satisfy

DOMAIN NEUTRALITY
- Do not assume any business domain, entity name, or field beyond what appears in ENDPOINT_PLACEHOLDER and CONDITIONS_PLACEHOLDER.

PARAMETER PLACEMENT (critical)
- GET / HEAD / DELETE: put non-path parameters in "query" unless the spec says they are headers. "payload" should almost always be {}.
- POST / PUT / PATCH: put body fields in "payload". Put query-style parameters in "query" if the spec places them in query.
- Path parameters: never invent new path keys; embed values in "endpoint" path segments.

COVERAGE (critical)
- Every condition id appearing in the input MUST appear in at least one test case's "covered_condition_ids".
- Prefer merging compatible valid conditions into fewer positive tests.
- For invalid / boundary-invalid cases, isolate the violating parameter when possible so expected_status and diagnosis stay clear.

EXPECTED STATUS
- Align with the OpenAPI responses section when possible. If ambiguous, use 400 for client validation errors and 404 for missing resources.

Output ONLY one JSON object (no markdown fences, no commentary).
